# Architecture

```
packages/
  emulate/          # npm CLI shim and native process programmatic API
  @emulators/
    core/           # compatibility helpers and native proxy facade
    adapter-next/   # Next.js App Router proxy integration
    vercel/         # Vercel API metadata and compatibility package
    github/         # GitHub API metadata and compatibility package
    google/         # Google metadata and compatibility package
    slack/          # Slack metadata and compatibility package
    apple/          # Apple metadata and compatibility package
    microsoft/      # Microsoft metadata and compatibility package
    aws/            # AWS metadata, compatibility package, and SDK conformance tests
    okta/           # Okta metadata package
    clerk/          # Clerk metadata package
    mongoatlas/     # MongoDB Atlas metadata package
    resend/         # Resend metadata package and SDK conformance tests
    stripe/         # Stripe metadata package
internal/
  core/             # Go HTTP, store, assets, config, and UI foundations
  services/         # Native Go service implementations
vercel/             # Native Go Vercel Function adapter
apps/
  web/              # Documentation site
```

## Native Runtime

The native Go runtime is the service engine for the CLI, the programmatic API, SDK conformance tests, compatibility facades, and Vercel Go Function previews. It serves selected services from one local server by default. `--portless` starts service-specific native listeners so each service can keep a named HTTPS alias.

## npm Packages

The `emulate` npm package provides the CLI shim, native binary resolution, and `createEmulator()` helper. Platform-specific optional packages carry the native binaries for macOS, Linux, and Windows.

The `@emulators/*` packages remain published for package discovery and compatibility, but they no longer contain Node.js service implementations. Service routing, state, UI, and protocol behavior live in Go.

## State

All local CLI state is held in memory with no database. State is populated from seed config on startup and can be modified via API calls during a test run.

Vercel Go Function previews use warm in-memory state by default. Cold starts reset to a fresh store, warm invocations reuse mutations, and concurrent function instances can diverge. For snapshots across cold starts, implement `vercel.Persistence` in `api/emulate.go`.
